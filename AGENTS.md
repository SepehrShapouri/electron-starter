# AGENTS.md

Operational guide for coding agents working in this repository.

## Project Snapshot

- Stack: Electron Forge + React 19 + TypeScript + TanStack Router + Tailwind CSS v4.
- Bundling: `rolldown-vite` via Electron Forge Vite plugin.
- Tests: Vitest (unit/integration) + Playwright (E2E).
- Package manager: npm (`package-lock.json` present).
- Path alias: `@/*` -> `src/*`.

## Setup

- Install deps: `npm install` (local dev) or `npm ci` (CI/reproducible).
- Start app (Electron dev): `npm start`.
- Node version: use modern Node (README says v18+, CI uses Node 20).

## Build, Lint, Typecheck, Test Commands

- `npm run lint` - runs ESLint on `src/`.
- `npm run type-check` - runs `tsc --noEmit`.
- `npm run format` - runs Prettier write mode on `src/`.
- `npm run format:check` - checks formatting on `src/`.
- `npm run test` - runs Vitest once (`vitest --run`).
- `npm run test:watch` - runs Vitest in watch mode.
- `npm run test:ui` - runs Vitest UI.
- `npm run test:coverage` - runs Vitest with coverage.
- `npm run test:e2e` - runs Playwright suite.
- `npm run test:e2e:ui` - Playwright UI mode.
- `npm run test:e2e:headed` - Playwright headed mode.
- `npm run routes:generate` - generates `src/routeTree.gen.ts`.
- `npm run build` - `tsc && electron-forge make`.
- `npm run package` / `npm run make` - package/distributables.

## Single-Test Recipes (Important)

- Single Vitest file: `npm run test -- src/lib/gateway-cron.test.ts`
- Single Vitest test name: `npm run test -- src/lib/gateway-cron.test.ts -t "formats daily cron expressions"`
- Vitest direct alternative: `npx vitest run src/lib/gateway-cron.test.ts -t "formats daily cron expressions"`
- Single Playwright file: `npx playwright test tests/e2e/example.spec.ts`
- Single Playwright test title: `npx playwright test tests/e2e/example.spec.ts --grep "navigation works"`
- Single Playwright browser: `npx playwright test --project=chromium tests/e2e/example.spec.ts`

## Recommended Local Validation Order

For feature work, run in this order unless asked otherwise:

1. `npm run type-check`
2. `npm run lint`
3. `npm run test` (or targeted tests during iteration)
4. `npm run test:e2e` for UI/flow-impacting changes

## Generated / Managed Files

- `src/routeTree.gen.ts` is generated by TanStack Router CLI.
- Do not hand-edit generated route tree files.
- After adding/removing route files, run `npm run routes:generate`.

## Code Style: Formatting

- Semicolons required.
- Single quotes for strings.
- Trailing commas: ES5 style.
- Max line width: 80.
- 2-space indentation, no tabs.
- Keep arrow parens minimal when possible (`arrowParens: avoid`).

## Code Style: TypeScript and Types

- TypeScript `strict` mode is enabled; preserve strict typing.
- `noUnusedLocals` and `noUnusedParameters` are enabled; remove dead code.
- Prefer explicit domain types (`type`/`interface`) for API payloads and events.
- Use `unknown` for untrusted data and narrow with runtime checks.
- Avoid `any`; ESLint warns on explicit `any`.
- Use `import type` for type-only imports when practical.
- Prefer union literal types for finite states (see update state/status patterns).

## Code Style: Imports

- Prefer `@/` alias for internal imports under `src`.
- Keep imports grouped: platform/framework, aliased internal modules, then relative imports.
- Keep type imports explicit where it improves clarity.
- Avoid deep relative paths when alias can express intent.

## Code Style: Naming and File Conventions

- React components: PascalCase names for component identifiers.
- Hooks: `useX` naming and keep them pure/idempotent.
- Utility modules and many route/page files use kebab-case filenames.
- Route files under `src/routes` map to URL structure; prefer lowercase route paths.
- Keep constants UPPER_SNAKE_CASE when they are static config keys.

## React / UI Conventions

- Functional components and hooks only (no class components).
- Prefer composition and small reusable UI primitives.
- Use `cn()` from `src/lib/utils.ts` for className merging.
- Keep Tailwind classes in JSX; avoid unnecessary CSS files unless shared/global.
- Use shadcn/Radix patterns already present in `src/components/ui`.

## Error Handling Conventions

- Wrap network errors with user-friendly `Error` messages.
- Use helper mappers like `getApiErrorMessage(error)` for Axios errors.
- For non-critical UI side effects, graceful fallback (`null`/`false`) is acceptable.
- Never swallow important errors silently unless intentionally non-blocking.
- Preserve actionable messages for IPC, auth, gateway, and update flows.

## Testing Conventions

- Unit tests live near code (`*.test.ts`) or under `src/test`.
- E2E tests live in `tests/e2e`.
- Vitest uses `jsdom` and setup file `src/test/setup.ts`.
- Prefer behavior-focused assertions via Testing Library.
- Keep tests deterministic; avoid timing-dependent flakes.

## Linting Notes

- React Hooks rules are enabled and should be treated as required.
- `react/react-in-jsx-scope` and `react/jsx-uses-react` are off (React 17+ style).
- Some shadcn files are intentionally ignored; avoid broadening ignores casually.

## Electron and IPC Notes

- Main-process logic in `src/main.ts`, preload bridge in `src/preload.ts`.
- Keep `contextIsolation: true`; do not enable `nodeIntegration` casually.
- Expose safe, minimal API surface via `contextBridge` (`window.electronAPI`).
- Validate payloads crossing IPC boundaries.

## Security / Secrets

- Never commit secrets, tokens, or signing credentials.
- Signing/notarization uses env vars in CI; keep them external.
- Be careful when editing auth/device identity/gateway handshake code.

## CI and Release Expectations

- CI workflow: `.github/workflows/build.yml`.
- CI runs type-check and lint (currently `continue-on-error: true`) before build.
- Builds and publishes artifacts to a separate releases repository.
- Do not change release automation unless explicitly requested.
